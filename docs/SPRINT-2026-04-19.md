## P1 — reliability gaps surfaced this week

### 4. Fix tunnel survival across LaunchAgent oneshot exit **(M, ~1 day)**
> **Problem:** `com.traylinx.tytus.plist` is `KeepAlive=false`. The oneshot launches `sudo tytus tunnel-up` via sudo + `setsid()`, reports success, exits. launchd appears to reap the detached tunnel daemon anyway. On macOS reboot, tunnels do not come back without manual `tytus connect`.
> **Acceptance:** after `launchctl bootout` / `bootstrap` cycle on the oneshot, the tunnel daemon's pidfile + `utun*` interface persist indefinitely. E2e harness run 30 minutes after fresh bootstrap still sees `P5 tunnel daemon alive`.

Two candidate approaches:
1. **Try `AbandonProcessGroup=true`** in the plist. Cheap test; Apple docs are ambiguous on whether launchd respects this for grandchildren. If it works we're done.
2. **Restructure as `KeepAlive=true` supervisor.** New plist runs a thin `tytus tunnel-supervise` that owns the boringtun child directly — no sudo-orphan chain. Bonus: network-loss recovery becomes automatic (supervisor re-runs `tunnel-up` on child exit). More work but also fixes suspend/resume cycles.

Recommend: try (1) first, escalate to (2) only if launchd still reaps.

### 5. Apple code signing for the CLI + tray app **(L, ~2 days)**
> **Problem:** unsigned binaries re-trigger keychain ACL approval on every update. Dialog is invisible to background LaunchAgents → silent failure. This is the #1 user-visible reliability issue and the dev-friction we hit all day today.
> **Acceptance:** `codesign --verify` passes on `tytus` and `Tytus.app`; after `sparkle`/installer-style update, existing keychain entries remain accessible without user interaction.

Steps (rough):
1. Buy Apple Developer membership ($99/yr).
2. Generate Developer ID Application cert, store in login keychain.
3. Add `codesign --deep --sign "Developer ID Application: ..."` step to the release flow for `tytus`, `tytus-mcp`, and `Tytus.app`.
4. Notarize DMG via `xcrun notarytool submit`.
5. Staple notarization ticket so it works offline.
6. Update install.sh / the DMG's drag-to-Applications flow to handle the existing-install upgrade path without triggering Gatekeeper.

Eats ~2 days when you factor in Apple's review queue on first notarization.

## P1.5 — things that will bite us at scale

### 6. Bake the Hermes dashboard SPA build into `tytus-hermes:latest` **(S, ~3h)**
> **Problem:** first boot of every Hermes pod runs `npm install && npm run build` (~45s). User clicks "Open in Browser" and sees a spinner for a minute. Worse: each restart re-runs it if the volume is fresh.
> **Acceptance:** `hermes dashboard` responds on 9119 within 3s of container start; no `npm install` in runtime logs.

Add to `infrastructure/hermes/docker/Dockerfile.pod`:
```dockerfile
# Pre-build the dashboard SPA so first-boot is fast
RUN cd /opt/hermes/web && npm install --omit=dev && npm run build
```
Test: fresh container hits 9119 returning the pre-built HTML immediately.

### 7. Update `llm-docs.md` for Hermes changes **(S, ~2h)**
> **Problem:** agents driving Tytus read `tytus llm-docs` as source of truth. It doesn't know:
> - Hermes now has a browser UI
> - Forwarder multiplexes on Hermes (`/v1/*` → gateway, else → dashboard)
> - `config.user.json` overlay for OpenClaw origins
> - `config.user.yaml` overlay for Hermes config
> - `is_logged_in` now tolerates missing RT with valid AT

> **Acceptance:** new sections covering the above; the existing "Agent deploy plan" section reconciles with real behavior.

Also propagate to the hosted `SKILL.md` at `raw.githubusercontent.com/.../skill.md`.

### 8. Structured observability — `/tmp/tytus/events.jsonl` **(M, ~1 day)**
> **Problem:** today's debugging required reading `autostart.log`, `daemon.log`, `tunnel-*.log`, `ui-*-forwarder.log` side-by-side. No way to answer "what happened in the last hour" programmatically.
> **Acceptance:** every state transition (connect/disconnect/revoke, token refresh outcome, forwarder start/stop, pairing auto-approve, upstream health change) emits a JSON line with `{ts, event, pod_id?, outcome, details}`. `tytus doctor --events 1h` renders the last hour as a timeline.

Implementation: one event writer in `core`, call sites across the lifecycle surfaces. Log rotation at 10 MB. Structured JSON, not tracing subscriber.

### 9. Rate-limit `/pod/agent/exec` **(S, ~4h)**
> **Problem:** the forwarder self-heal hits this on every cold start. A misbehaving forwarder (tight reconnect loop, crash loop) can DoS Provider. `/pod/agent/exec` is also our most powerful endpoint — an audit log is overdue.
> **Acceptance:** Provider rejects >5 exec calls/minute per pod with 429. Every accepted call is written to an audit table with `{client_id, pod_id, command_hash, exit_code, ts}`.

Owner: wannolot-provider repo.

## P2 — error-path hardening

### 10. Actionable error states in `tytus connect` / `tytus status` **(M, ~1 day)**
> **Problem:** current error UX is mostly generic strings. A real user seeing "Authentication expired" vs "Provider 502" vs "WG rekey failed" vs "agent crash-looping" gets no clear next action.
> **Acceptance:** a small enum `ConnectError` (PROVIDER_DOWN, POD_ALLOCATING, AGENT_UNHEALTHY, TUNNEL_FLAPPING, AUTH_EXPIRED) threads through. Each variant has a templated message + a "next step" line. `tytus doctor` can diagnose each.

### 11. Error recovery: auto-reconnect on WG rekey failure **(M, ~1 day)**
> **Problem:** WG tunnels occasionally drop after the 3-hour rekey window if the peer's handshake is lost. Currently manual `tytus disconnect && tytus connect` is needed.
> **Acceptance:** tunnel monitor detects handshake-stale condition (no successful rekey in 5 min when traffic flowing) and auto-reconnects with exponential backoff.

Related to sprint item #4's supervisor model — might fold together.

## P3 — nice-to-have

### 12. Windows `wintun.dll` bundling **(L, ~3 days)**
> **Acceptance:** `tytus` on Windows 10/11 runs boringtun via wintun without manual DLL install. Distribution: MSI or chocolatey package.

Several paper cuts expected (path separators, keychain→DPAPI, launchd→Task Scheduler).

### 13. `tytus backup` / `tytus restore` **(S, ~3h)**
> **Acceptance:** `tytus backup > backup.json` writes everything needed to restore state on a new Mac (except keychain secrets, which must be re-entered via `tytus login`). `tytus restore backup.json` reconstructs state.json and re-registers tunnels.

### 14. Scalesys HPA maxReplicas > 1 **(L, ~2-3 days)**
> **Problem:** SQLite backend blocks horizontal scaling. Not urgent while we're single-digit customers; becomes urgent at ~50 active pods.
> **Acceptance:** Scalesys on Postgres; `fleet.db` migration script; k8s HPA unblocked.
